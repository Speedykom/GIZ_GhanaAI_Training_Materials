1. The “Time Machine” for Code

As Web developers, you likely know Git. But in AI Engineering, version control is not just about saving code—it is about reproducibility, experiment tracking, and managing your professional portfolio.

Git vs. GitHub: The Distinction

It is crucial to separate the tool from the service:

  • Git: The engine. A distributed version control system that runs locally on your machine. It tracks changes, manages history, and handles branching.
  • GitHub: The host. A cloud platform that hosts Git repositories and adds collaboration tools (Pull Requests, Issues, Actions).

Alternatives

While we will use GitHub exclusively for this program (as it is the industry standard for AI portfolios), you will often see these alternatives in enterprise environments:

  • GitLab: Popular for its built-in CI/CD pipelines.
  • Bitbucket: Often used by companies heavily invested in the Atlassian stack (Jira/Confluence).
ImportantWhy GitHub for this Accelerator?

European employers want to see your “green squares” (activity graph) and code quality on GitHub. It is your primary resume.

Key Terminology

Term Definition
Repository (Repo) A project folder containing your code and the .git hidden folder (the history).
Commit A snapshot of your project at a specific moment. Like a “save point” in a game.
Stage (Index) The holding area for changes you want to include in the next commit.
Branch A parallel version of your repository. Use branches to experiment without breaking the “main” code.
Remote A version of your repository hosted on the internet (e.g., on GitHub).
Clone Downloading a repository from a Remote to your local machine.
Fork Creating a personal copy of someone else’s project on GitHub so you can modify it.
Pull Request (PR) Proposing changes from your branch to the main codebase.

2. Installing Git

We will use the command line for all Git operations to ensure you understand the underlying mechanics.

🖥️ Windows (via Winget)

Open PowerShell as Administrator and run:

winget install --id Git.Git -e --source winget

🍎 macOS

If you installed Xcode command line tools, you likely have it. To install/update via Homebrew:

brew install git

🐧 Linux (Ubuntu)

sudo apt-get update
sudo apt-get install git

⚙️ First-Time Configuration

You must tell Git who you are. This information is baked into every commit you make.

git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

3. Setting Up Your GitHub Account

Before we can push code to the cloud, you need a secure home for your projects.

Step 1: Sign Up

The easiest way to get started is using your primary email address.

  1. Go to github.com/signup.
  2. Enter your email: We recommend using your primary Gmail (or similar personal email) so your portfolio travels with you, regardless of where you work.
  3. Create a Password: Make it strong.
  4. Choose a Username: Pick something professional (e.g., dev-richard is better than coolguy123). This will be part of your portfolio URL.
  5. Solve the Puzzle: Verify you are human.

Step 2: Enable 2-Factor Authentication (2FA) 🛡️

This is mandatory. GitHub now requires 2FA for many contributors to improve supply chain security. If you lose access to your account, you lose your portfolio.

  1. Once logged in, click your Profile Photo (top right) → Settings.
  2. In the left sidebar, click Password and authentication.
  3. Under “Two-factor authentication”, click Enable two-factor authentication.
  4. Recommended Method: Use an Authenticator App (like Google Authenticator or Microsoft Authenticator) on your phone.
    • Scan the QR code shown on the screen.
    • Enter the 6-digit code from your phone.
  5. Download Recovery Codes: Save these in a safe place (not just on your laptop). If you lose your phone, these are the only way to get back in.

4. The Workflow: From Folder to Cloud

Let’s walk through the lifecycle of a repository.

Scenario A: Starting a New Project

1. Initialize (git init)

Turn a regular folder into a Git repository.

mkdir my-ai-project
cd my-ai-project
git init
  • What it does: Creates a hidden .git folder. This folder tracks all changes.

2. Check Status (git status)

See what Git “sees”.

git status
  • What it does: Shows which files are changed, new (untracked), or staged.

3. Stage Changes (git add)

Move files from “Working Directory” to “Staging Area”.

git add .
  • The . flag: Adds all files in the current directory.
  • Specific file: git add main.py (Better practice if you only want to commit specific changes).

4. Commit (git commit)

Save the snapshot.

git commit -m "Initial commit: Set up project structure"
  • The -m flag: Allows you to write the commit message inline. Without it, Git opens a default text editor (often Vim) which can be confusing for beginners.

5. Create Remote Repo

Go to GitHub.com, click the + icon, and create a new repository. Do not initialize it with a README or .gitignore yet (keep it empty).

6. Link & Push (git remote, git push)

Connect your local folder to the cloud.

git remote add origin https://github.com/YOUR_USERNAME/REPO_NAME.git
git push -u origin main
  • git remote add origin [URL]: Creates a shortcut named “origin” that points to your GitHub URL.
  • git push: Uploads your commits.
  • The -u flag: Stands for “upstream”. It links your local main branch to the remote main branch so in the future you can just type git push without arguments.

Scenario B: Connecting Existing Code to a New Repo

Often, you start coding before thinking about Git. Here is how to sync an existing project folder to GitHub.

  1. Create the blank repo on GitHub (as above).
  2. Navigate to your project folder in terminal.
  3. Run the connection sequence:
git init
git add .
git commit -m "First commit of existing code"
git branch -M main
git remote add origin https://github.com/YOUR_USERNAME/REPO_NAME.git
git push -u origin main

Why are these steps necessary?

  • git branch -M main: GitHub recently changed the default branch name from master to main. Your local Git might still default to master. This command renames your branch to match GitHub’s standard, preventing “divergent branch” errors.
  • remote add: Your local folder has no idea the internet exists. You must explicitly introduce it to the GitHub server address.

5. Critical Pitfalls in AI Engineering

As you move from Web Development to AI, the stakes with Git get higher. Here are the most common mistakes to avoid.

❌ 1. Committing API Keys (The #1 Sin)

In AI, we use expensive APIs (OpenAI, Anthropic).

  • The Mistake: Hardcoding keys like api_key = "sk-12345..." and pushing to GitHub.
  • The Consequence: Bots scan GitHub constantly. Your key will be stolen and maxed out in seconds.
  • The Fix: Use .env files and strictly add .env to your .gitignore file.

❌ 2. Committing Massive Datasets

  • The Mistake: Adding a 500MB .csv or .images folder to Git.
  • The Consequence: Git is built for text, not massive binary files. Your repo will become slow and GitHub will reject the push (100MB limit).
  • The Fix: Add data folders (e.g., data/) to .gitignore. Store data in the cloud (S3, Google Drive) or download it via script.

❌ 3. “Detached HEAD” State

  • The Scenario: You checkout a specific commit hash (to see old code) and then start coding.
  • The Fix: Always create a branch if you want to make changes from an old point: git checkout -b new-feature-branch.

❌ 4. The “Vim Trap”

  • The Scenario: You run git commit without -m and get stuck in a screen you can’t exit.
  • The Fix: Type :q! and hit Enter to escape without saving. Then use git commit -m "message".
